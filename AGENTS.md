# 專案開發協作規則

## 核心原則

在開發新功能、修改既有功能、調整 UI/UX、變更資料結構或重構程式碼之前，必須先充分理解使用者的需求與預期結果，並主動思考是否存在更好的設計或實作方式。

## 開發前確認

開始實作前，請先向使用者清楚說明：

1. 對需求的理解，以及預計解決的問題。
2. 預計修改或新增的功能、檔案與範圍。
3. 重要的假設、可能影響的既有行為，以及需要使用者決定的事項。
4. 建議採用的方案與理由。
5. 若有更佳的替代方案，說明各方案的優缺點、成本、風險與適用情境。
6. 可能被忽略的例外情況、錯誤情境、權限／安全性、效能、維護性與使用體驗問題。

除非是非常明確、低風險且不會改變產品行為的微小修正，否則在使用者確認方案前，不要直接開始實作。若需求有多種合理解讀，應先提出具體問題，不要自行猜測關鍵產品決策。

## 提出建議

不要只照字面執行需求。理解需求後，應主動檢查：

- 是否有更簡單、穩定、可維護或可擴充的做法。
- 是否能沿用現有元件、樣式、資料格式或程式架構。
- 是否會造成既有功能回歸、資料遺失或使用者操作困惑。
- 是否需要補充載入中、空資料、錯誤、取消、重複操作與行動裝置等狀態。
- 是否有更適合的互動流程、資訊架構或介面呈現方式。

提出建議時要說明原因與取捨；若建議會改變原始需求的範圍，必須先取得使用者同意。

## 實作與驗證

使用者確認後再進行實作，並遵守以下流程：

1. 只修改已確認的範圍；若過程中發現需要擴大範圍，先停下來說明並重新確認。
2. 優先配合現有專案結構與風格，避免不必要的重寫。
3. 實作完成後執行適當的測試、檢查或手動驗證。
4. 回報完成內容、修改的檔案、驗證結果、尚未處理的風險與建議的後續工作。

## 溝通方式

以清楚、具體、可決策的方式溝通。確認需求時，應讓使用者能快速回答「要採用哪個方案」或「是否同意目前方案」，避免只提出籠統的問題。若資訊已足夠，直接做合理且低風險的判斷；若缺少的資訊會影響功能方向、資料安全或使用者體驗，則必須先確認。
